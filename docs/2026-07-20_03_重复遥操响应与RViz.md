# 重复遥操响应与 RViz

本文记录 `teleop-session --repeat` 的重复会话提示、采集停止边界、ROS 收尾和 RViz 重启约定。本项目只调整自身编排层与 YAML，不修改 `pika_ros` 的代码或 launch 文件。

## 重复会话提示

“已按现场情况完成基站校准”只在一次 `--repeat` 命令的第一条轨迹前要求空格确认。第二条及后续轨迹不会重复此提示，但仍会执行：

1. `preflight`；
2. 工作区安全确认；
3. YAML 配置的起始位姿；
4. Pika 传感器、控制器和两路 RViz 的重新启动；
5. Sense 确认、采集、停止与保存。

基站、频道或硬件发生变化时，必须先退出当前会话并显式执行 `calibrate-base --mode force`，不能把跳过重复提示当作跳过校准。

## 停止采集

动作完成提示处按空格后，编排层立刻设置停止信号。采集器会在相机读取、JPEG 编码、状态对齐和 HDF5 入队前后检查该信号；若按键发生在一轮取帧中间，该轮会被丢弃，不会作为尾帧写入轨迹。已经在按键前进入 HDF5 队列的样本会保留，保证已采集数据不被截断。

终端先显示“已禁止新帧进入采集队列”，随后在相机和机器人采样线程完全停止后显示“采集已停止”。之后的等待仅用于关闭 HDF5、生成 `quality.json`、`manifest.json` 与校验和，不再采样新数据。

## ROS 收尾

实体设备侧双击 Sense 停止遥操后按第二次空格。本项目会先提示正在关闭 ROS，再同时向传感器和控制器的独立进程组发送 `SIGINT`；若仍有进程存活，才同时升级为 `SIGTERM`、`SIGKILL`。并行收尾避免过去控制器和传感器各自等待、总等待时间相加造成的终端卡顿。

## RViz 重启

每一条会话都由已有 Pika launch 启动 Sense RViz 与 Piper RViz。以前第一条数据采集导入 OpenCV 后，父进程可能留下：

```text
QT_QPA_PLATFORM_PLUGIN_PATH=.../.venv/.../cv2/qt/plugins
```

第二条 ROS/RViz 继承该路径后会加载不兼容的 Qt `xcb` 插件，并在日志出现 `rviz2 ... exit code -6`。本项目现在在启动 ROS 子进程时清除 `QT_QPA_PLATFORM_PLUGIN_PATH` 与 `QT_PLUGIN_PATH`，令 RViz 使用 ROS 系统 Qt 插件。

两路 RViz 都是启动就绪条件；任一 `rviz2` 退出都会报告启动失败，不会再显示“遥操节点已就绪”。此时查看该条目录下 `teleop_logs/<时间戳>/pika_sense.log` 和 `piper_controller.log`，修复显示环境后重新开始本条会话。
