# Pika Sense 遥操采集流程

本流程将已有 Pika Sense 遥操与本工程的数据采集编排在一起。遥操 ROS 工程、传感器脚本和 Piper 采集器仍是三个独立边界：编排层只负责启动顺序、进程生命周期、人工确认和数据保留决策。

## 一次性准备

1. 初始化子模块并安装依赖：

   ```bash
   git submodule update --init --recursive
   uv sync --extra dev --extra realsense
   ```

   若默认 uv 缓存目录不可写，可使用：

   ```bash
   uv --cache-dir /tmp/uv-cache sync --extra dev --extra realsense
   ```

2. 从 `configs/piper_d405_d435.example.yaml` 和 `configs/pika_sense_piper.example.yaml` 各复制一份不提交的现场 YAML。填写相机序列号、标定、采集任务和采集员哈希；确认 `robot.can_name` 与实际 CAN 一致。
3. 检查现场 YAML 的 `robot.tool_offset_m`。当前已有 ROS launch 使用 `tcp_offset=[0, 0, 0.13, 0, 0, 0]`，但该数值是否等于实物工具相对法兰的偏移必须经现场标定确认。采集配置与遥操控制配置必须使用同一物理 TCP 定义。

## 基站校准

首次使用本机、增减基站或切换频道时执行强制校准：

```bash
uv run piper-collect calibrate-base \
  --teleop-config configs/现场_pika_sense.yaml \
  --mode force
```

定位漂移或使用中移动过基站时执行诊断：

```bash
uv run piper-collect calibrate-base \
  --teleop-config configs/现场_pika_sense.yaml \
  --mode diagnose
```

这两个命令只封装需求中给出的 `survive-cli` 调用；其交互过程和校准结论仍由现场人员判断。

## 单条与连续采集

执行单条遥操采集：

```bash
uv run piper-collect teleop-session \
  --config configs/现场_piper.yaml \
  --teleop-config configs/现场_pika_sense.yaml
```

需要连续采多条时追加 `--repeat`。每条完成后先选择保存或删除，再回答是否继续；下一条会重新经过完整安全确认，不会继承上条遥操状态。

```bash
uv run piper-collect teleop-session \
  --config configs/现场_piper.yaml \
  --teleop-config configs/现场_pika_sense.yaml \
  --repeat
```

实际顺序如下：

```text
基站校准完成确认
  -> 只读 preflight
  -> 人工将 Piper 安全移至起始位姿并确认
  -> 前台 sudo -v
  -> 启动 Pika Sense 传感器进程
  -> 启动 Piper ROS 遥操进程
  -> 人工双击 Sense 夹爪并确认遥操已启用
  -> 启动 HDF5 数据采集
  -> 人工先停止遥操，再在编排终端按 Enter
  -> 关闭两个 ROS 进程组并完成 HDF5
  -> 选择保存或删除本条数据
```

`preflight` 是只读检查，不会移动机械臂。起始位姿也不由本工程发送：必须由获授权的现场人员使用已有安全流程完成。采集器只会在“已双击 Sense 夹爪启用遥操”的确认之后启动，因此起始对齐数据不会混入遥操未启用阶段。

结束时必须先在实体设备侧停止遥操，再按 Enter。按 Enter 会请求采集器完成写入、随后终止两个 ROS 启动命令所属的进程组。它不是急停手段；发生危险时应立即使用机械臂和现场规定的硬件急停流程。

会话期间 ROS 输出写入采集输出根目录下的 `teleop_logs/<时间戳>/`。保存时，轨迹目录包含 `trajectory.hdf5`、`quality.json`、`manifest.json` 和 `checksums.sha256`；删除会移除该条轨迹目录，不影响其他日期或轨迹。

## 遥操代码边界

未修改以下既有遥操代码：

- `~/pika_ros/scripts/start_single_sensor_whit_teleop.bash`
- `pika_remote_agx_arm/launch/teleop_single_piper.launch.py`

编排层仅通过 YAML 中的 shell 命令启动它们。传感器脚本包含 `sudo`，因此示例配置先执行 `sudo -v`，避免后台遥操进程与采集确认争用终端输入。

当前 launch 文件固定了 `can0`、`auto_enable=true`、`effector_type=agx_gripper` 和 0.13 m 的 Z 向 `tcp_offset`。现有需求不需要修改它，但更换 CAN、夹爪、工具或希望自动识别 Sense 双击状态时，应该在遥操工程中增加配置参数和一个明确的“遥操已激活”状态话题。编排层目前故意采用人工确认，避免在没有可靠 ROS 状态契约时错误提前采集。

## 验证

无硬件环境完成依赖安装后执行：

```bash
uv run pytest
uv run piper-collect collect --config configs/mock_piper.yaml
```

真实设备首次接入时，先独立运行 `preflight`，再执行一次短遥操轨迹并检查 `quality.json`、RGB 画面、关节变化和 TCP 时间同步。
