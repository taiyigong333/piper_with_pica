# Pika Sense 遥操采集流程

本流程将已有 Pika Sense 遥操与本工程的数据采集编排在一起。遥操 ROS 工程、传感器脚本和 Piper 采集器仍是三个独立边界：编排层只负责启动顺序、进程生命周期、人工确认和数据保留决策。

## 一次性准备

1. 初始化子模块并安装依赖：

   ```bash
   git submodule update --init --recursive
   uv sync --extra dev --extra realsense
   ```

   若默认 uv 缓存目录不可写，可使用：

   ```bash
   uv --cache-dir /tmp/uv-cache sync --extra dev --extra realsense
   ```

   本项目由 `.python-version` 与 `pyproject.toml` 固定为 uv 管理的 Python 3.10。该 `.venv` 不会修改系统 Python 或 `pika` Conda 环境；ROS 子进程由遥操 YAML 中的 `conda activate pika` 提供 Python 3.10 依赖。

2. 从 `configs/piper_d405_d435.example.yaml` 和 `configs/pika_sense_piper.example.yaml` 各复制一份不提交的现场 YAML。填写相机序列号、标定、采集任务和采集员哈希；确认 `robot.can_name` 与实际 CAN 一致。
3. 检查现场 YAML 的 `robot.tool_offset_m`。当前已有 ROS launch 使用 `tcp_offset=[0, 0, 0.13, 0, 0, 0]`，但该数值是否等于实物工具相对法兰的偏移必须经现场标定确认。采集配置与遥操控制配置必须使用同一物理 TCP 定义。
4. 配置 `robot.initial_pose`。只有 `enabled: true` 时，会话才在启动 ROS 前移动 Piper：`mode: joint` 使用 `joint_positions_rad`（六个 rad），`mode: tcp` 使用 `tcp_pose`（`[x,y,z,rx,ry,rz]`，m + rad）。先执行以下只读命令记录目标，再填写 YAML；首次使用保持 `speed_percent: 10`：

   ```bash
   uv run piper-collect read-piper-state --config configs/现场_piper.yaml
   ```

## 基站校准

首次使用本机、增减基站或切换频道时执行强制校准：

```bash
uv run piper-collect calibrate-base \
  --teleop-config configs/pika_sense_piper.example.yaml \
  --mode force
```

定位漂移或使用中移动过基站时执行诊断：

```bash
uv run piper-collect calibrate-base \
  --teleop-config configs/pika_sense_piper.example.yaml \
  --mode diagnose
```

这两个命令只封装需求中给出的 `survive-cli` 调用；其交互过程和校准结论仍由现场人员判断。

## 单条与连续采集

执行单条遥操采集：

```bash
uv run piper-collect teleop-session \
  --config configs/现场_piper.yaml \
  --teleop-config configs/现场_pika_sense.yaml
```

需要连续采多条时追加 `--repeat`。每条默认保存；保存后按空格才会开始下一条，按 `q` 结束会话。下一条会重新经过完整安全检查和起始位姿控制，不会继承上条遥操状态。

```bash
uv run piper-collect teleop-session \
  --config configs/现场_piper.yaml \
  --teleop-config configs/现场_pika_sense.yaml \
  --repeat
```

实际顺序如下：

```text
基站校准完成后按空格
  -> 只读 preflight
  -> 确认工作区安全后按空格
  -> 根据 robot.initial_pose 低速移动 Piper 并校验到位
  -> 前台 sudo -v
  -> 启动 Pika Sense 传感器进程
  -> 启动 Piper ROS 遥操进程
  -> 检查关键 ROS 节点日志均正常
  -> 双击 Sense 夹爪，确认 Piper 实际跟随后按空格
  -> 启动 HDF5 数据采集
  -> 动作完成后按空格，停止采集并完成 HDF5 写盘
  -> 人工双击 Sense 停止遥操，再按空格
  -> 优雅关闭两个 ROS 进程组
  -> 默认保存本条数据
```

`preflight` 是只读检查，不会移动机械臂。起始位姿控制只由 `teleop-session` 在 `robot.initial_pose.enabled=true` 时执行，使用 Piper SDK 的 CAN 位置控制；它以配置速度周期下发同一目标，读取反馈进入容差后立刻断开控制连接。若使能或到位超时，会切换为待机模式并拒绝启动遥操。TCP 目标使用物理工具中心点，控制时会按同一个 `tool_offset_m` 反算 Piper 法兰目标。

结束时先按空格停止采集并完成写入，保持遥操运行；看到写盘完成提示后，再在实体设备侧双击 Sense 停止遥操并按第二次空格。第二次空格会优先向 ROS 进程组发送 `SIGINT`，等待 launch 按正常流程关闭节点，必要时才升级为 `SIGTERM` 和 `SIGKILL`。它不是急停手段；发生危险时应立即使用机械臂和现场规定的硬件急停流程。

会话期间 ROS 输出写入采集输出根目录下的 `teleop_logs/<时间戳>/`。控制器日志必须包含 `agx_arm_ctrl_single`、`arm_ik_pose_node.py`、`pub_delta_pose.py` 与 Piper RViz 的启动记录；出现 `Traceback`、`ModuleNotFoundError` 或任一关键节点/RViz 死亡记录会在采集前失败。传感器日志中的 Sense RViz 退出同样会阻止会话；若出现 `xcb` 或 `exit code -6`，检查 Qt 环境变量隔离是否生效。传感器日志若出现 sudo 无终端/需密码也会失败，先在前台完成 `sudo -v`。保存时，轨迹目录包含 `trajectory.hdf5`、`quality.json`、`manifest.json` 和 `checksums.sha256`；仅显式使用 `--on-complete delete` 才会删除该条轨迹目录。残留进程清理与轨迹验证命令见 `2026-07-20_01_遥操收尾与轨迹检查.md`。

## 遥操代码边界

未修改以下既有遥操代码：

- `~/pika_ros/scripts/start_single_sensor_whit_teleop.bash`
- `pika_remote_agx_arm/launch/teleop_single_piper.launch.py`

编排层仅通过 YAML 中的 shell 命令启动它们。传感器脚本包含 `sudo`，因此示例配置先执行 `sudo -v`。编排层还会为两个 ROS 子进程移除 uv 的 `VIRTUAL_ENV`、`PYTHONHOME` 与 `PYTHONPATH`，避免 `uv run` 的解释器污染 `#!/usr/bin/env python3`。随后由遥操 YAML 的 `conda activate pika` 选择已有 Python 3.10 环境；这不是对 Pika ROS 代码的修改。

当前 launch 文件固定了 `can0`、`auto_enable=true`、`effector_type=agx_gripper` 和 0.13 m 的 Z 向 `tcp_offset`。现有需求不需要修改它，但更换 CAN、夹爪、工具或希望自动识别 Sense 双击状态时，应该在遥操工程中增加配置参数和一个明确的“遥操已激活”状态话题。当前仍以“Piper 已实际跟随 Sense 后按空格”的人工确认守住采集起点。

## 验证

无硬件环境完成依赖安装后执行：

```bash
uv run pytest
uv run piper-collect collect --config configs/mock_piper.yaml
```

真实设备首次接入时，先独立运行 `preflight`，再执行一次短遥操轨迹并检查 `quality.json`、RGB 画面、关节变化和 TCP 时间同步。
