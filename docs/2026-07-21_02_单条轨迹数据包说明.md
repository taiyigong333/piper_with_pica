# 单条轨迹数据包说明

本文说明一条已经正常完成的轨迹目录中各文件和 HDF5 字段的含义。示例目录为：

```text
/home/ubuntu/gcj/projects/data_collect/data/corm_in_plate/real/20260720/real_piper_20260720T210847_0001
```

该目录代表一次完整的真实设备采集，不是单独的一张图片或单条关节反馈。目录层级的含义如下：

| 路径部分 | 含义 |
| --- | --- |
| `corm_in_plate` | 当前 YAML 中的 `session.output_root`，即本任务的数据根目录。 |
| `real` | `metadata/data_type`，表示真实设备数据；另外允许 `sim` 或 `synthetic`。 |
| `20260720` | 采集开始当天，格式为 `YYYYMMDD`。 |
| `real_piper_20260720T210847_0001` | 轨迹 ID。默认格式为 `<data_type>_<robot_name>_<YYYYMMDDTHHMMSS>_0001`；若在 YAML 中显式设置 `session.trajectory_id`，则以配置值为准。 |

## 目录中的交付文件

```text
real_piper_20260720T210847_0001/
├── trajectory.hdf5
├── quality.json
├── manifest.json
└── checksums.sha256
```

| 文件 | 含义 | 使用方式 |
| --- | --- | --- |
| `trajectory.hdf5` | 主数据文件：图像、时间轴、关节角、TCP、夹爪反馈、设备和采集配置快照。 | 训练读取、`validate` 和 `trajectory-viewer` 的输入。 |
| `quality.json` | 自动质检报告：结构、时间轴、图像解码、重复帧及静态标定项检查结果。 | 先看 `result`、`errors`、`warnings` 和 `manual_review`。 |
| `manifest.json` | 交付清单：轨迹 ID、格式版本、完成状态、质量结论、设备列表，以及主文件的 SHA-256。 | 用于确认目录是否是完整交付包。 |
| `checksums.sha256` | `trajectory.hdf5` 与 `quality.json` 的 SHA-256 校验表。 | 在该目录执行 `sha256sum -c checksums.sha256`，检查传输或复制后是否被改动。 |

只有存在正式 `trajectory.hdf5` 的目录才是已完成轨迹。若只有 `trajectory.hdf5.partial`，表示写入未正常收尾，不能通过改名将其当成有效数据。

## 本示例的实际内容

对示例目录进行只读检查得到：

| 项目 | 实际值 | 含义 |
| --- | --- | --- |
| `metadata/trajectory_length` | `204` | 204 个对齐多相机观测时刻，也是每个 `*_align` 序列的长度。 |
| RGB 相机 | `camera_front`、`camera_wrist_right` | 分别是前视 D435 和腕部 D405；两路均有 204 帧。 |
| 原始机器人反馈 | `465` 条 | 以机器人反馈频率保存，不与相机帧按数组下标一一对应。 |
| 已启用模态 | `rgb, arm_joint_positions, tcp_pose, gripper_position` | 本条未启用深度，因此没有 `camera_observations/depth_images`。 |
| TCP 表示 | `xyz_rxryrz` | 每行是 `[x, y, z, rx, ry, rz]`，位置单位为 m，姿态单位为 rad。 |
| 自动质检 | `pass` | 没有结构性错误；仍有未配置相机外参和 `base_to_robot` 的告警，不能据此直接做跨坐标系变换。 |

## HDF5 顶层字段

`trajectory.hdf5` 使用 HDF5 分组保存。下表中的 `/` 表示 HDF5 根目录。

| 分组 | 内容 |
| --- | --- |
| `/metadata` | 任务指令、采集时间、数据类型、采集员标识、格式版本、轨迹长度和 TCP 表示。 |
| `/robot_model`、`/end_effector_model` | 机器人名称和末端/夹爪驱动名称。 |
| `/camera_model`、`/camera_color_resolution`、`/camera_color_channel` | 每台相机的型号、`[宽, 高]` 与颜色通道约定。 |
| `/camera_intrinsics` | 运行时读出的相机内参：`matrix` 为 $3 \times 3$ 内参矩阵，`dist_coeffs` 为畸变参数。 |
| `/camera_extrinsics` | 可选的采集参考 `base -> camera` 齐次变换；未标定时该分组可为空。 |
| `/base_to_robot_transformation` | 可选的采集参考 `base -> Piper` 齐次变换；未标定时此分组不存在。 |
| `/acquisition_config` | 写入时使用的采集速率、对齐策略、最大状态年龄、模态开关和 YAML 文件名快照。 |
| `/camera_observations` | 以多相机组帧为基准的时间轴和编码后的相机数据。 |
| `/puppet` | Piper 的原始反馈序列，以及与相机时间轴逐帧对齐的反馈序列。 |

### 元数据和采集配置

常用字段如下：

| 字段 | 含义 |
| --- | --- |
| `/metadata/language_instruction` | 该条轨迹的自然语言任务指令。本示例为 `pick up the corn and put it on the plate`。 |
| `/metadata/data_type` | 数据来源类型，本例为 `real`。 |
| `/metadata/data_format_version` | 数据格式版本；读取程序应据此判断字段和单位兼容性。 |
| `/metadata/collection_time` | 写入开始时记录的本地时间。 |
| `/metadata/collector` | YAML 中填写的采集员标识。 |
| `/metadata/pose_representation` | TCP 行向量的表示方式。当前 Piper 使用 `xyz_rxryrz`；旧兼容格式可能为 `xyz_xyzw`。 |
| `/metadata/trajectory_length` | 对齐观测数量，不等于原始机器人反馈数量。 |
| `/acquisition_config/robot_state_hz` | 原始机器人反馈目标频率，本示例为 60 Hz。 |
| `/acquisition_config/camera_rig_hz` | 多相机组帧目标频率，本示例为 30 Hz。 |
| `/acquisition_config/max_alignment_age_ms` | 相机组帧可使用的最新机器人反馈最大年龄，本示例为 100 ms。超过此阈值的组帧会被丢弃，不写入对齐序列。 |
| `/acquisition_config/alignment_strategy` | 当前为 `latest_robot_state_with_max_age`，即选取组帧时最新且未过期的机器人反馈。 |

## 相机观测：一行就是一个训练时刻

`/camera_observations` 是训练和人工检查应优先使用的主时间轴。索引 `i` 表示同一时刻的完整多相机观测：

| 字段 | 形状 | 每个索引 `i` 的含义 |
| --- | --- | --- |
| `/camera_observations/timestamp` | `(N,)` | 第 `i` 组相机图像完成读取、准备发布对齐快照时的 Unix 时间戳，单位为秒。 |
| `/camera_observations/is_intervene` | `(N,)` | 与该时刻关联的人工干预标记。当前 Piper 只读适配器没有独立干预信号，本示例 204 项均为 `false`；不能将 `false` 解释成“操作者没有移动机械臂”。 |
| `/camera_observations/color_images/camera_front` | `(N,)` | 前视 D435 的第 `i` 帧 RGB 图像，存为可变长度的 JPEG 字节。 |
| `/camera_observations/color_images/camera_wrist_right` | `(N,)` | 腕部 D405 的第 `i` 帧 RGB 图像，存为可变长度的 JPEG 字节。 |
| `/camera_observations/depth_images/<camera>` | `(N,)`，可选 | 仅在 YAML 启用深度时存在；每项为 `uint16` PNG 字节。本示例未启用。 |

因此，对齐训练样本 `i` 应同时取两路图像和下文所有 `*_align/data[i]`。不同相机在同一 `i` 上共享时间轴，不需要按文件时间另行匹配。

## 机器人反馈：`raw` 和 `align` 的区别

`/puppet` 中每一个状态分组都包含三个同长度字段：

| 子字段 | 含义 |
| --- | --- |
| `timestamp` | Unix 时间戳，单位为秒。 |
| `is_intervene` | 与该条反馈关联的干预标记；当前 Piper 轨迹通常全为 `false`。 |
| `data` | 实际读取的机器人反馈数值，不是下发给 Piper 的控制目标。 |

每种状态都有两套序列：

| 后缀 | 采样方式 | 与图像的关系 | 推荐用途 |
| --- | --- | --- |
| `_raw` | 按 `robot_state_hz` 写入每一条可读取的真实反馈。 | 长度通常大于相机帧数，不能用相同数组下标对应图像。 | 精细分析机器人反馈频率、延迟、抖动。 |
| `_align` | 每取得一组完整相机帧，复制当时最新且未过期的反馈，并把时间戳设为该相机组帧时间。 | 长度严格等于 `trajectory_length`，索引 `i` 与所有相机第 `i` 帧一一对应。 | 浏览器查看、视觉-状态训练和逐帧人工检查。 |

本示例中，`*_raw` 长度为 465，`*_align` 长度为 204。不要将第 100 条 `raw` 状态当作第 100 帧图像的状态；需要图像配对时使用 `align`。

### 关节角

| 字段 | 形状 | `data` 的顺序和单位 |
| --- | --- | --- |
| `/puppet/arm_single_position_raw/data` | `(M, 6)` | `[J1, J2, J3, J4, J5, J6]`，均为 rad。 |
| `/puppet/arm_single_position_align/data` | `(N, 6)` | 与相机第 `i` 帧对齐的六关节反馈，均为 rad。 |

### TCP 位姿

| 字段 | 形状 | `data` 的顺序和单位 |
| --- | --- | --- |
| `/puppet/end_effector_single_pose_raw/data` | `(M, 6)` | 原始 TCP 反馈。 |
| `/puppet/end_effector_single_pose_align/data` | `(N, 6)` | 与相机第 `i` 帧对齐的 TCP 反馈。 |

必须先读取 `/metadata/pose_representation` 再解释 TCP 列。对于本示例的 `xyz_rxryrz`，每行固定为 `[x, y, z, rx, ry, rz]`：`x/y/z` 的单位为 m，`rx/ry/rz` 是 Piper 原生欧拉角，单位为 rad。若元数据为 `xyz_xyzw`，则宽度为 7，列顺序改为 `[x, y, z, qx, qy, qz, qw]`，不能按六维欧拉角读取。

### 夹爪

| 字段 | 形状 | `data` 的含义 |
| --- | --- | --- |
| `/puppet/end_effector_single_position_raw/data` | `(M, 1)` | 夹爪读取到的开闭行程，单位为 m。 |
| `/puppet/end_effector_single_position_align/data` | `(N, 1)` | 与相机第 `i` 帧对齐的夹爪行程，单位为 m。 |

该值来自 Piper 反馈，不是“打开/关闭”的布尔标签，也不是夹爪控制命令。零值只能表示当前读数为零，不能单独用于判断夹爪是否成功夹住物体。

## 质量和训练前检查

`quality.json` 的 `result: pass` 说明自动结构检查没有发现错误，不等价于任务动作一定合格。训练前至少检查：

1. `quality.json` 的 `errors` 必须为空，`warnings` 中的标定缺失是否影响当前用途。
2. `manual_review.status` 是人工质检状态；本示例为 `pending`，表示尚未完成人工确认。
3. `collection_stats` 中的 `camera_bundles` 应等于 `trajectory_length`；`skipped_camera_bundles_*`、`robot_read_errors` 和 `camera_read_errors` 应结合任务判断是否可接受。
4. 使用 `trajectory-viewer` 逐帧检查两路图像与 `*_align` 的关节、TCP、夹爪变化是否一致。
5. 复制或传输数据后在目录内执行校验和检查：

```bash
sha256sum -c checksums.sha256
```

本工程只记录相机图像和 Piper 的实际反馈，不记录 Pika Sense 的控制指令或 Piper 的下发控制命令。因此，`*_align` 可作为图像对应的机器人状态观测，但不能被误认为已经包含了可直接监督学习的控制动作标签。
