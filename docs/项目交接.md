# 项目交接

更新时间：2026-07-20

## 已确定的设计

1. 工程根目录为 `code_piper`，其本身是独立 Git 仓库。
2. `piper_sdk` 必须以 Git 子模块存在；当前固定于 `c05c5454b1cf61c05ad26385e0c0a3aa6d3c7bad`。
3. Python 环境统一使用 uv，并通过 `.python-version`/`pyproject.toml` 固定为 Python 3.10；项目依赖的 Piper SDK 由子模块本地安装，不修改系统 Python 或 `pika` Conda 环境。
4. 数据格式遵守仓库上级的《数据采集与留存规范》：异步 HDF5、raw/align、图像编码与包级留存文件。
5. Piper TCP 默认采用原生 `xyz_rxryrz`，即位置 `m` 加欧拉角 `rad`；内部四元数仅用于计算工具偏移。`xyz_xyzw` 保留为兼容格式，读取数据必须以轨迹元数据中的 `pose_representation` 为准。
6. Pika Sense 遥操与采集通过 `teleop_session.py` 编排，未修改现有 ROS 遥操脚本。启动 ROS 子进程前会清除 uv 的 Python 环境变量，并由 YAML 激活已有的 `pika` Python 3.10 环境；同时检查关键节点启动日志。会话收尾固定为“结束采集和写盘，再双击停止 Sense，最后关闭 ROS”；关闭时按 `SIGINT`、`SIGTERM`、`SIGKILL` 逐级回收独立进程组。完整现场流程见 `docs/2026-07-18_02_Pika_Sense遥操采集流程.md`。
7. `robot.initial_pose` 为遥操会话新增的显式控制边界：仅 `enabled=true` 时、且在 ROS 遥操启动前发送 Piper 起始位姿命令；支持六关节 rad 或原生 TCP `xyz_rxryrz`（m + rad），低速到位后立即断开。`preflight` 与常规 `collect` 仍严格只读。
8. `piper-collect read-piper-state --config <现场 YAML>` 与 `scripts/read_piper_state.py` 可一次性只读输出六关节和 TCP，适合记录起始位姿配置。
9. `piper-collect teleop-status` 只读列出残留的本项目 Pika 遥操进程；只有显式执行 `piper-collect teleop-stop --terminate` 才会逐级结束它们。轨迹检查流程见 `docs/2026-07-20_01_遥操收尾与轨迹检查.md`。
10. `piper-collect trajectory-viewer --config <采集 YAML>` 提供零额外 Web 依赖的本地只读轨迹界面。它递归查找 `session.output_root` 下的已完成轨迹，显示 RGB、对齐关节/TCP/夹爪、时序曲线和 `quality.json`；默认仅监听 `127.0.0.1`，不控制硬件。
11. `docs/使用说明.md` 是不带日期的长期操作入口；每次功能、配置、命令或流程变更都必须维护它，并新增相应的日期文档。
12. `docs/` 中的技术文档以 `YYYY-MM-DD_序号_中文主题.md` 命名，保证按文件名即可按时间排序；`使用说明.md` 与 `项目交接.md` 不参与该排序。

## 现场接入前必须完成

- 用 `ip link` 确认 Piper 对应的 SocketCAN 名称和 1 Mbps 配置；本工程不配置 CAN 接口。
- 用 `piper-collect discover-realsense` 记录 D405/D435 序列号，并填入现场 YAML。
- 测量或标定 `base_to_robot`、每台 `base_to_camera` 和工具偏移；不要用单位矩阵代替未知标定。
- 在完全无运动的状态下执行 `preflight`，检查 TCP 和六个关节值量级、相机分辨率与深度对齐。
- 使用 `read-piper-state` 记录目标后，再填写 `robot.initial_pose`；首次启用保持 `speed_percent: 10`，确认路径、限位和工具偏移后才设 `enabled: true`。
- 首次采集后执行 `validate`，再人工抽查 RGB、关节、TCP 的同步性。
- 对照 ROS launch 中的 `can0` 与 `tcp_offset=[0, 0, 0.13, 0, 0, 0]`，确认采集 YAML 的 CAN 与 `tool_offset_m` 代表同一实际机器人和 TCP。

## 风险与待验证项

- Piper SDK 的末端反馈姿态使用 SDK 文档所述欧拉角。当前落盘保留原生欧拉角；工具偏移计算的临时四元数转换采用 SDK TCP 示例一致的旋转顺序，现场须用已知姿态进行方向验证。
- D405/D435 能否同时以配置中的流参数启动依赖 USB 控制器带宽和实际固件；失败时降低分辨率或分布到不同控制器，但需重新预检。
- 腕部相机外参会随工具、相机支架变化而变化；每次变更必须新建标定配置并随轨迹写入。
- 会话编排能通过日志拦截 `rclpy`/`casadi` 导入失败和关键节点死亡，但仍不能读取 Sense 双击后的“遥操已启用”状态；当前以“Piper 已实际跟随后按空格”的人工确认守住采集起点。若需要自动化，必须先由遥操工程提供稳定状态话题或服务契约。
- RViz 单独退出不代表遥操核心失败；只检查 `agx_arm_ctrl_single`、IK、增量位姿、locator 和串口节点。若发生会话异常，先运行 `teleop-status`，清理确认过的残留进程后才重启下一条会话。
- 在本受限执行环境中，Piper SDK 内部调用 `ip` 读取 CAN 位速率会被 netlink 权限阻断；现场正常终端可执行 `read-piper-state`。已在允许 CAN 访问的上下文验证 Python 3.10 下的真实关节角、TCP、夹爪与预检读取正常。
