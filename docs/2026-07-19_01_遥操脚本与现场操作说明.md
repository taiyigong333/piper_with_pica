# 遥操脚本与现场操作说明

本文说明本项目与既有 `pika_ros` 的脚本边界、每个命令的作用和现场启动顺序。项目 Python 依赖使用 uv 管理的 Python 3.10；`pika` Conda 环境仅供既有 ROS 遥操进程使用，两者不能混用安装依赖。

## 文件与职责

| 位置 | 作用 | 是否向 Piper 发送控制帧 |
| --- | --- | --- |
| `scripts/read_piper_state.py` | 可直接运行的状态读取包装脚本，等价于 CLI 的 `read-piper-state`。 | 否 |
| `src/taiyi_piper_collect/piper_state.py` | 读取上述脚本的实际实现，输出六关节角和 TCP。 | 否 |
| `src/taiyi_piper_collect/teleop_session.py` | 编排校准确认、预检、起始位姿、ROS 进程、空格交互和 HDF5 采集。 | 仅在启用 `robot.initial_pose` 时 |
| `src/taiyi_piper_collect/devices/piper_motion.py` | 仅负责起始位姿控制；到位、失败或超时后断开控制连接。 | 是 |
| `configs/piper_d405_d435.example.yaml` | 采集、TCP 表示和起始位姿模板。 | 否 |
| `configs/pika_sense_piper.example.yaml` | 已有 Pika Sense 与 Piper ROS 命令的启动模板。 | 否 |
| `~/pika_ros/scripts/start_single_sensor_whit_teleop.bash` | 已有 Pika Sense 传感器与定位启动脚本。 | 由既有 ROS 工程决定 |
| `pika_remote_agx_arm/launch/teleop_single_piper.launch.py` | 已有 Piper ROS 遥操 launch 文件。 | 由既有 ROS 工程决定 |

最后两项位于 `pika_ros`，本项目只通过 YAML 启动它们，**不会修改其代码或 launch 文件**。

## 命令清单

所有命令均在项目根目录执行。`UV_CACHE_DIR=/tmp/uv-cache` 只指定 uv 缓存目录；项目环境仍是本仓库的 `.venv`。

| 命令 | 作用 | 是否移动 Piper | 何时使用 |
| --- | --- | --- | --- |
| `uv run piper-collect discover-realsense --require-device` | 列出已连接的 RealSense 序列号。 | 否 | 首次填写相机配置或更换相机后 |
| `uv run piper-collect preflight --config configs/现场_piper.yaml` | 读取一帧相机、Piper、夹爪反馈，检查配置和设备是否匹配。 | 否 | 每次真实采集或遥操前 |
| `uv run piper-collect read-piper-state --config configs/现场_piper.yaml` | 输出六关节角 `joint_positions_rad` 与当前 TCP。 | 否 | 记录 `robot.initial_pose` 目标前 |
| `uv run python scripts/read_piper_state.py --config configs/现场_piper.yaml` | 与上一条完全等价，适合需要直接调用脚本的场景。 | 否 | 同上 |
| `uv run piper-collect calibrate-base --teleop-config configs/现场_pika_sense.yaml --mode force` | 清除并重新进行 Pika 基站定位校准。 | 否 | 首次使用、增减基站、切换频道或移动基站后 |
| `uv run piper-collect calibrate-base --teleop-config configs/现场_pika_sense.yaml --mode diagnose` | 启动 Pika 定位诊断，不替代强制校准。 | 否 | 怀疑定位漂移时 |
| `uv run piper-collect teleop-session --config configs/现场_piper.yaml --teleop-config configs/现场_pika_sense.yaml` | 执行一条完整遥操采集会话。 | 取决于 `robot.initial_pose.enabled` | 已完成校准和预检后 |
| 在上一条末尾加 `--repeat` | 每条默认保存，按空格开始下一条独立轨迹。 | 同上 | 连续采集多条时 |

`read-piper-state` 的 TCP 格式由 `session.pose_representation` 决定。Piper 现场配置应为 `xyz_rxryrz`，输出顺序为 `[x, y, z, rx, ry, rz]`，位置单位 m、角度单位 rad。

## 起始位姿配置

`robot.initial_pose` 默认不启用，避免模板被复制后意外驱动机械臂。启用前必须先阅读当前状态、确认工作空间和路径安全。

关节角方式：

```yaml
robot:
  initial_pose:
    enabled: true
    mode: joint
    joint_positions_rad: [0.0, 0.2, -0.4, 0.0, 0.3, 0.0]
    tcp_pose: null
    speed_percent: 10
    timeout_s: 30
```

TCP 方式：

```yaml
session:
  pose_representation: xyz_rxryrz
robot:
  tool_offset_m: [0.0, 0.0, 0.13]
  initial_pose:
    enabled: true
    mode: tcp
    joint_positions_rad: null
    tcp_pose: [0.25, 0.00, 0.22, 3.14, 0.0, 0.0]
    speed_percent: 10
    timeout_s: 30
```

两种方式只填写当前 `mode` 对应的目标。TCP 是物理工具中心点：程序使用同一份 `tool_offset_m` 将其换算为 Piper SDK 所需的法兰末端目标。到位条件由 `joint_tolerance_rad`，或 `position_tolerance_m` 与 `orientation_tolerance_rad` 控制。超时会切换 Piper 为待机模式，且不会启动 ROS 遥操。

不要把 Piper SDK 示例中的静态关节范围当作现场固件的唯一真值：本机实际 J5 反馈可超出该示例范围。会话不再用该静态表拒绝实测起始姿态，最终机械限位由 Piper 固件执行；因此只能将已通过 `read-piper-state` 读取、且已确认安全的姿态写入 YAML。

## 一条遥操轨迹的实际顺序

1. 确认没有旧的 Pika 定位或遥操进程占用 Sense；需要时完成 `force` 校准。
2. 执行 `preflight`，仅当 `result` 是 `pass` 才继续。
3. 启动 `teleop-session` 后，终端依次等待空格：校准完成、工作区安全、Sense 双击后 Piper 已实际跟随、实体设备侧已停止遥操。
4. 若 `robot.initial_pose.enabled=true`，第二次按空格后才低速移动 Piper 到 YAML 目标；到位才启动 Pika ROS 进程。
5. ROS 日志通过检查后，双击 Sense 夹爪。确认 Piper 已实际跟随后按空格，HDF5 采集才开始。
6. 完成动作时先在实体设备侧停止遥操，再按空格。会话关闭 ROS 进程并完成写盘，默认保存轨迹。

按 `q` 可在任一空格确认阶段取消。会话不再要求输入 `yes`、`no` 或 Enter。`--on-complete delete` 是唯一会删除已写轨迹的显式选项。

## ROS Python 环境

项目 uv 环境和 ROS 的依赖用途不同：

- `uv run piper-collect ...` 使用项目 `.venv` 的 Python 3.10，负责采集、Piper SDK 和会话编排。
- `controller.command` 在启动 ROS 前 source `~/pika_ros/install/setup.bash`，再执行 `conda activate pika`；该环境必须能够导入 `rclpy` 与 `casadi`。
- 编排层会从 ROS 子进程移除 `VIRTUAL_ENV`、`PYTHONHOME`、`PYTHONPATH` 与 uv 的 `.venv/bin` 路径，避免 `#!/usr/bin/env python3` 错误选择项目解释器。

可以用以下只读命令检查 Pika ROS 解释器和依赖；不启动 ROS 节点：

```bash
source ~/pika_ros/install/setup.bash
source ~/miniconda3/etc/profile.d/conda.sh
conda activate pika
python -c 'import sys, rclpy, casadi; print(sys.executable); print(sys.version)'
```

输出应指向 `~/miniconda3/envs/pika/bin/python`，且版本为 Python 3.10。不要向系统 Python、项目 `.venv` 或 `pika_ros` 源码补装 ROS 依赖。

## 常见现象

| 现象 | 原因 | 处理 |
| --- | --- | --- |
| `LIBUSB_ERROR_BUSY (-6)` 或 `Could not claim interface ... Tracker` | Sense Tracker 的 USB 接口已被已有定位器、遥操进程或其他驱动占用。 | 从原终端结束旧 Pika 会话或定位程序，确认没有并行的 `survive-cli`/ROS 定位进程后重新执行校准。后续仍出现时不要视为可靠校准完成。 |
| 控制器日志有 `ModuleNotFoundError: casadi` 或 `_rclpy_pybind11` | ROS 节点使用了错误的 Python 解释器，或 `pika` 环境缺依赖。 | 按上一节检查 `pika` 环境；会话会在开始采集前报告失败。 |
| 传感器日志有 `sudo: a terminal is required` | `start_single_sensor_whit_teleop.bash` 的 sudo 权限未缓存。 | 在会话主终端完成 `sudo -v`；不要让后台脚本读取密码。 |
| 出现两个 RViz，但 Sense 双击后 Piper 不动 | RViz 启动不代表 `arm_ik_pose_node.py` 和 `pub_delta_pose.py` 存活。 | 查看本条轨迹的 `teleop_logs/<时间戳>/piper_controller.log`；确认没有 traceback、导入失败或节点死亡记录。 |
| `read-piper-state` 提示 CAN 位速率检查失败 | 当前终端没有读取 SocketCAN/netlink 的权限，或 `can0` 未按 1 Mbps 启动。 | 在现场正常终端执行 `ip -details link show can0`；确认状态为 `UP` 且 `bitrate 1000000`，然后重试只读命令。 |

更多采集格式说明见 `2026-07-18_01_架构与数据契约.md`，会话状态机见 `2026-07-18_02_Pika_Sense遥操采集流程.md`。
