# 遥操收尾与轨迹检查

本文规定 Pika Sense 遥操采集的结束顺序、异常后的残留进程处理，以及已保存轨迹的检查方法。它不修改 `pika_ros` 代码或 launch 文件。

## 一条轨迹的结束顺序

会话中的空格确认分为两个不同阶段，不能合并：

```text
Sense 遥操运行且 HDF5 正在采集
  -> 当前动作完成，按空格
  -> 采集器停止采样、关闭 trajectory.hdf5、写 quality.json/manifest.json/checksums.sha256
  -> 双击 Sense 夹爪停止实体遥操，按空格
  -> 编排层向 ROS 进程组发送 SIGINT 并等待退出
  -> 必要时升级为 SIGTERM、SIGKILL
```

因此，第一次空格只结束**数据采集**，不会停止 Sense 或 ROS 程序。只有看到“数据采集和写盘已完成”后，才双击 Sense 结束遥操并按第二次空格。这样轨迹文件已经完整关闭，不会把停止遥操前后的空白时间混入数据。

发生机械臂危险时仍应使用现场硬件急停；上述空格和进程终止不是急停手段。

## 残留进程与重启

正常会话关闭优先向独立 ROS 进程组发送 `SIGINT`，使 `ros2 launch` 有机会按 Ctrl-C 流程回收其节点。等待超时后才发送 `SIGTERM`，最后才发送 `SIGKILL`。这比直接杀启动 shell 更能避免 `agx_arm_ctrl_single`、`pika_single_locator_node` 等节点残留。

在开始新一条遥操前，先检查当前用户是否仍有可明确识别的 Pika 单臂遥操进程：

```bash
# 只列出残留进程，不改变任何进程状态。
UV_CACHE_DIR=/tmp/uv-cache uv run piper-collect teleop-status
```

如果输出的 `process_count` 不为 `0`，先显式清理，再重新检查：

```bash
# 不带 --terminate 时只预览；带该参数才向匹配的 Pika 进程发送终止信号。
UV_CACHE_DIR=/tmp/uv-cache uv run piper-collect teleop-stop --terminate
UV_CACHE_DIR=/tmp/uv-cache uv run piper-collect teleop-status
```

`teleop-stop` 只匹配当前用户命令行中可识别的本项目 Pika 单臂遥操进程，包括 controller、locator、IK、增量位姿、串口节点和对应的 `ros2 launch`。不要使用宽泛的 `pkill -f ros2`，它可能误杀无关 ROS 任务。

控制器日志中单独的 RViz 退出不等于遥操核心失败。会话只会把 `agx_arm_ctrl_single`、`arm_ik_pose_node.py`、`pub_delta_pose.py`，以及传感器的 locator/串口节点死亡视为失败。若会话失败，先检查本条轨迹的 `teleop_logs/<时间戳>/piper_controller.log` 和 `pika_sense.log`，然后执行 `teleop-status`。

## 轨迹自动产物

一次正常保存的轨迹目录包含：

| 文件 | 作用 | 产生时机 |
| --- | --- | --- |
| `trajectory.hdf5` | RGB、机器人关节/TCP/夹爪、时间轴和元数据。 | 采集器正常关闭后从 `.partial` 原子改名。 |
| `quality.json` | 自动结构、时间轴、数据形状、图像解码和重复帧检查。 | `trajectory.hdf5` 写完后。 |
| `manifest.json` | 轨迹 ID、设备信息、格式版本和质量结论。 | `quality.json` 写完后。 |
| `checksums.sha256` | HDF5 与质量报告的 SHA-256 校验和。 | manifest 生成时。 |

若目录中只存在 `trajectory.hdf5.partial`，说明采集没有完成正常收尾；不要把它当作可交付轨迹，也不要通过重命名伪造完成文件。

## 轨迹检查命令

将 `<trajectory_path>` 替换为某条 `trajectory.hdf5` 的绝对路径。以下命令均只读，不会控制 Piper、启动相机或修改轨迹：

```bash
# 按现场 YAML 检查已启用模态、HDF5 结构、时间轴、TCP 宽度、图像可解码性和重复帧比例。
UV_CACHE_DIR=/tmp/uv-cache uv run piper-collect validate <trajectory_path> \
  --config configs/pick_up_the_corn_and_put_it_on_the_plate.yaml
```

`result=pass` 表示没有结构性错误；`warnings` 仍需要人工判断，例如未配置外参、图像重复帧比例过高。`result=fail` 时不要继续使用该轨迹，应先保留日志和原始文件用于定位问题。

随后验证交付文件未被修改：

```bash
# 在该轨迹目录执行；仅验证 HDF5 与 quality.json 的 SHA-256。
cd <trajectory_directory>
sha256sum -c checksums.sha256
```

人工检查时至少打开 `quality.json`，确认 `collection_stats`、相机重复帧告警和 `manual_review.status`；再抽查 HDF5 中 RGB、关节和 TCP 是否随动作同步变化。`manifest.json` 的 `quality_status` 必须与 `quality.json` 的 `result` 一致。

## 推荐重启流程

```text
teleop-status 为 0
  -> 必要时 force 校准
  -> preflight 为 pass
  -> teleop-session
  -> 第一次空格结束采集并完成写盘
  -> 双击停止 Sense，第二次空格关闭 ROS
  -> validate + sha256sum 检查本条轨迹
  -> repeat 时才开始下一条
```
